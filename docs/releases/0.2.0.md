# FerrisSearch 0.2.0

Release branch: `release/0.2.0`

Comparison base: `v0.0.1..ad33b68`

This release covers 81 commits across 95 files, with 57,435 insertions and 3,275 deletions compared to `v0.0.1`.

FerrisSearch 0.2.0 is the release where the project makes a clear jump from basic search engine to distributed search-and-analytics system. Compared to `v0.0.1`, this release adds distributed SQL, much deeper fast-field execution, stronger replication and cluster behavior, better tooling, and a more credible performance story.

## What Ships in 0.2.0

- FerrisSearch now handles far more than keyword search. The engine picked up vector search, typed field mappings, wildcard, prefix, and fuzzy queries, richer aggregations, field sorting with stable tie-breaking, and dynamic settings.
- SQL is now a first-class feature instead of a side path. This release adds the global SQL endpoint, `ferris-cli`, `EXPLAIN`, `EXPLAIN ANALYZE`, distributed grouped partial aggregation, distributed fast-field SQL over Arrow IPC, multi-column `GROUP BY`, same-index semijoins, quoted hyphenated index support, case-insensitive source-column canonicalization, and streamed SQL execution.
- Performance moved closer to the intended architecture. The release range adds FAST-field execution improvements, `LIMIT` pushdown, `count(*)` fast paths, selective `_id` and `_score` reads, a column cache, batch numeric aggregation, direct scan and flat-array grouped aggregation, parallel segment scans, StringArena-based string handling, deferred JSON materialization, pre-sized coordinator maps, and lazy SQL batch streaming across the coordinator and transport.
- Cluster and replication behavior are much stronger than in `v0.0.1`. This includes WAL sequence ownership, ISR checkpoint tracking, replica recovery from the translog, replica promotion, index UUID isolation, `_count`, better refresh and flush fan-out, `INITIALIZING` shard visibility, authoritative join snapshots, and harder guarantees around node lifecycle and membership changes.
- Operability and developer workflow improved as well. The range adds Prometheus metrics, optional feature-gated gRPC TLS, CI automation, broader REST and replication coverage, more benchmark and ingestion scripts, and a much larger documentation surface around architecture, benchmarks, dynamic settings, and vector search.

## Why This Release Matters

- Queries that used to fall back to generic post-processing can now stay on search-aware execution paths more often, especially for grouped analytics and fast-field SQL.
- The CLI is now a serious workflow tool, not just a thin wrapper around HTTP calls. It supports interactive SQL usage, `EXPLAIN ANALYZE`, and streamed result consumption.
- The cluster is safer under restart, recovery, and join scenarios because shard identity, WAL sequencing, and membership changes are handled much more explicitly.
- The project now has the supporting infrastructure to back up the engine work: better tests, better benchmarks, better docs, and a real release narrative for users evaluating the system.
